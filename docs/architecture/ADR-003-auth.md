# ADR-003: Аутентификация и авторизация

**Дата:** 2026-05-15
**Статус:** Принято

## Контекст

Система управления закупками требует разграничения доступа. Разные роли пользователей должны иметь разные права.

## Решение

**JWT + bcrypt** для аутентификации.

### Роли
| Роль | Описание | Права |
|------|----------|-------|
| admin | Администратор | Полный доступ |
| lab_head | Заведующий КДЛ | Чтение/запись, согласование заявок |
| lab_tech | Лаборант | Чтение, создание заявок, расход со склада |
| economist | Экономист | Чтение, работа с КП и ценами |

### Flow
1. Пользователь логинится → получает JWT access token (24ч)
2. Каждый запрос содержит `Authorization: Bearer <token>`
3. Middleware проверяет токен и извлекает user_id/role
4. Endpoints проверяют роль через `Depends(require_role)`

### Хранение паролей
- bcrypt через passlib
- Пароль никогда не хранится в открытом виде

## Альтернативы

- **Session-based**: проще, но не масштабируется для API
- **OAuth2 + Keycloak**: избыточно для внутренней системы
- **Telegram Login**: удобно для лаборантов, но зависит от Telegram

## Последствия

- ✅ Простота реализации
- ✅ Stateless — не нужна сессия в Redis
- ⚠️ Нет refresh token (v0.1 — только access token 24ч)
- ⚠️ Logout невозможен без blacklist (приемлемо для v0.1)
